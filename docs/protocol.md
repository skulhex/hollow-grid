# Сетевой протокол

Документ фиксирует первый стабильный контракт между клиентом Hollow Grid и
сервером мультиплеера. Сервер считается авторитетным источником состояния:
клиенты отправляют действия, сервер проверяет их по текущему состоянию матча,
применяет допустимые действия и рассылает снимки состояния.

Контракт описывает сетевой формат данных. Внутренние возможности движка правил
не обязаны становиться публичными клиентскими командами.

## Игроки

Текущие идентификаторы игроков:

- `player_1`
- `player_2`

В снимках состояния игроки передаются в стабильном порядке матча:

```json
["player_1", "player_2"]
```

## Клетка

Клетки поля задаются axial-координатами hex-grid:

```json
{ "q": -2, "r": 0 }
```

Поля `q` и `r` обязательны и должны быть целыми числами.

## Сообщение действия

Клиент отправляет одно сообщение `Action` на каждое запрошенное действие игрока.

```json
{
  "type": "place_node",
  "player": "player_1",
  "cell": { "q": -2, "r": 0 }
}
```

Поля:

- `type`: обязательная строка. Значение должно входить в публичный список разрешённых действий.
- `player`: обязательная строка. Идентификатор игрока, который отправляет действие.
- `cell`: необязательная клетка. Обязательна для действий с целевой клеткой. Для `skip` должна отсутствовать.
- `source_cell`: необязательная клетка. Обязательна только для `striker_attack` и `hacker_hack`.

Действия, для которых обязательна `cell`:

- `place_node`
- `repair_node`
- `upgrade_harvester`
- `upgrade_striker`
- `upgrade_defender`
- `upgrade_hacker`
- `build_connection_module`
- `build_repair_module`
- `striker_attack`
- `hacker_hack`

Действия, для которых обязательны и `source_cell`, и `cell`:

```json
{
  "type": "striker_attack",
  "player": "player_1",
  "source_cell": { "q": -1, "r": 0 },
  "cell": { "q": 0, "r": 0 }
}
```

```json
{
  "type": "hacker_hack",
  "player": "player_2",
  "source_cell": { "q": 1, "r": 0 },
  "cell": { "q": 0, "r": 0 }
}
```

Действие завершения хода:

```json
{
  "type": "skip",
  "player": "player_1"
}
```

Для `skip` нельзя передавать `cell` или `source_cell`.

## Публичные действия клиента

Сервер принимает от клиентов только следующие типы действий:

- `place_node`
- `repair_node`
- `upgrade_harvester`
- `upgrade_striker`
- `upgrade_defender`
- `upgrade_hacker`
- `build_connection_module`
- `build_repair_module`
- `striker_attack`
- `hacker_hack`
- `skip`

Любой тип действия вне этого списка сервер должен отклонять до передачи в
движок правил.

## Внутренние действия

В локальном движке правил сейчас есть дополнительные действия:

- `break_node`
- `clear_node`

Они остаются внутренними и в текущем сетевом контракте не являются командами,
доступными игроку. Сервер не должен принимать `break_node` и `clear_node` из
клиентских сообщений, даже если локальный движок правил умеет их разобрать и
применить.

## Снимок состояния

Сервер отправляет снимок состояния `Snapshot` после создания матча и после
каждого принятого действия. Снимок должен содержать полное публичное состояние
матча, нужное клиенту для отрисовки и продолжения игры.

```json
{
  "players": ["player_1", "player_2"],
  "current_player": "player_1",
  "turn": 1,
  "round": 1,
  "core_hp": {
    "player_1": 5,
    "player_2": 5
  },
  "resources": {
    "player_1": 1,
    "player_2": 1
  },
  "action_limits": {
    "connection_actions_left": 1,
    "repair_actions_left": 1
  },
  "objects": [
    {
      "cell": { "q": -3, "r": 0 },
      "type": "core",
      "owner": "player_1",
      "active": true,
      "disabled": false
    },
    {
      "cell": { "q": -2, "r": 0 },
      "type": "node",
      "owner": "player_1",
      "active": true,
      "disabled": false,
      "role": "conduit",
      "ready": false,
      "action_charges": 0
    },
    {
      "cell": { "q": -2, "r": 1 },
      "type": "module",
      "owner": "player_1",
      "active": true,
      "disabled": false,
      "module_kind": "connection",
      "ready": false
    }
  ],
  "finished": false,
  "status_message": "Player 1: build, repair, upgrade, or end turn"
}
```

Поля `Snapshot`:

- `players`: упорядоченный список игроков, сейчас `["player_1", "player_2"]`.
- `current_player`: игрок, чей ход сейчас активен.
- `turn`: монотонный счётчик состояния или применённых действий. Это не номер раунда.
- `round`: текущий номер раунда.
- `core_hp`: HP ядер по идентификаторам игроков.
- `resources`: ресурсы по идентификаторам игроков.
- `action_limits`: оставшиеся лимиты действий активного игрока.
- `objects`: публичный список объектов на поле.
- `finished`: `true`, если матч завершён.
- `status_message`: последняя человекочитаемая строка статуса.

Поля `action_limits`:

- `connection_actions_left`: оставшиеся действия соединения активного игрока.
- `repair_actions_left`: оставшиеся действия восстановления активного игрока.

Каждый объект содержит:

- `cell`: клетка объекта.
- `type`: одно из значений `core`, `node`, `module`.
- `owner`: идентификатор владельца.
- `active`: участвует ли объект в активной сети и правилах.
- `disabled`: отключён ли объект.

Объекты `node` дополнительно содержат:

- `role`: одно из значений `conduit`, `harvester`, `striker`, `defender`, `hacker`.
- `ready`: готова ли роль после upkeep владельца.
- `action_charges`: оставшиеся заряды действия роли.

Объекты `module` дополнительно содержат:

- `module_kind`: одно из значений `connection`, `repair`.
- `ready`: прошёл ли модуль upkeep владельца и может ли дать эффект.

Объекты `core` не содержат полей роли или модуля.

## Проверка сообщений

- Сервер проверяет и форму сообщения, и игровые правила.
- Сообщение с корректной формой всё равно может быть отклонено правилами матча.
- Переданный клиентом `player` должен совпадать с ожидаемым активным игроком и владельцем соединения или сессии.
- Клиент должен считать `status_message` текстом для отображения, а не машинно-читаемым кодом результата.
